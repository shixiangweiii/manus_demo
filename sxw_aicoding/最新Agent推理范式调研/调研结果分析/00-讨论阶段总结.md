## 已确认的认知

1. **项目当前在 L4（Multi-Agent）层完全空白**——OrchestratorAgent 是单线程 LLM 循环，无独立子 agent 机制

2. **当前架构的根本限制是"上下文隔离"**——子任务的全部中间过程堆积在父 agent 的 _messages，ContextManager 只能"截断"不能"隔离"

3. **四个引擎共享同一配置体系（两个正交维度）**——规划调度层（PLAN_MODE 等）和执行引擎层（ENABLE_REACT_ENGINE_V2），新范式以特性开关叠加

4. **ENABLE_REACT_ENGINE_V2=true 独立解决了 ReAct 循环的消息竞态**（每次 execute() 创建局部 messages），但 ExecutorAgent 实例级共享状态问题仍需串行退化修复

5. **子 Agent 机制的三个工程价值**：上下文隔离、按子任务选引擎、实例级并行安全

## 未决的开放问题

1. **子 Agent 是横切基础设施还是 L4 新范式？** 影响引入时机和设计深度
2. **与 CodeAct 的优先级关系？** 两者正交但实现精力有限
3. **最小 MVP 是什么？** 是一个极简的 spawn_subagent tool 还是更完整的框架
4. **递归必要性？** 单层委派是否已足够
5. **成本控制？** 子 Agent 意味着更多 LLM 调用
