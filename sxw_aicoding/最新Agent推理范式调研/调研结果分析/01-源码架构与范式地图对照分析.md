# 源码架构与范式地图对照分析

> 分析日期：2026-05-13
> 依据：调研报告 01/02/03 + 项目源码（v8.0）
> 目的：对照调研报告中 5 层范式地图与项目源码现状，识别架构层面的真实约束和演化机会

---

## 一、项目当前架构的执行模型本质

无论使用哪个引擎（simple / DAG / emergent / goal-driven），底层执行模型是同一个：

```
LLM 生成 → 选工具 → 执行 → 观察结果 → 继续 LLM → ... → 最终答案
                    ↑
            全部中间过程堆积在 agent 的 self._messages 里
```

四个引擎的区别仅在**规划调度层**（怎么拆步骤、怎么编排），执行层是同一个单线程 ReAct 循环。具体源码映射：

| 引擎 | 规划方式 | 执行入口 | 消息管理 |
|------|---------|---------|---------|
| v1 simple | `PlannerAgent.create_plan()` → 扁平 Step 列表 | `ExecutorAgent._react_loop()` | BaseAgent._messages（线性累积） |
| v2 DAG | `PlannerAgent.create_dag()` → 三层 TaskDAG | `DAGExecutor._run_node()` → 共享 `ExecutorAgent` | 同一个 ExecutorAgent 实例的 _messages |
| v5 emergent | `EmergentPlannerAgent` 内部动态维护 TodoList | 内置 `_execute_todo_with_react()` | BaseAgent._messages（v5 专用实例） |
| v8 goal-driven | `GoalDrivenPlannerAgent` 目标文档 + 逆向规划 | 内置 ReAct 循环 + 目标反思 | BaseAgent._messages（v8 专用实例） |

### 关键源码约束

1. **OrchestratorAgent 只创建一个 ExecutorAgent 实例**（`orchestrator.py:117-121`），DAG 路径所有节点共享该实例的 `_messages`，导致并行竞态（已通过串行退化修复，见 `config.py:45 DAG_SERIAL_EXECUTION`）

2. **所有 agent 共享一个 LLMClient 实例**（`orchestrator.py` 构造函数），token 追踪集中化。子 agent 如需独立 token 追踪需额外处理。

3. **工具集全局共享**——所有 agent 使用同一组 `tools` 实例，`ToolRouter` 按 node_id 做失败统计隔离

4. **v6 ReActEngine**（`react/engine.py`）每次 `execute()` 创建局部 `messages` 列表（:112），不共享实例状态——这本身解决了消息竞态，但仅当 `ENABLE_REACT_ENGINE_V2=true` 时生效

## 二、5 层范式地图与项目现状的精准对照

```
                    ┌──────────────────────────────────────┐
   L5 Self-Evolve   │ MUSE / Voyager / Memp                │  ← 完全空白
                    ├──────────────────────────────────────┤
   L4 Multi-Agent   │ Chain-of-Agents / Claude Subagent    │  ← 完全空白
                    │ Manus 多角色 / ChatDev              │     Orchestrator 是单线程编排
                    ├──────────────────────────────────────┤
   L3 Reasoning     │ o1 / R1 Long CoT / LATS (MCTS)      │  ← v8 有线性反思（ReflAct）
                    │ ReflAct (状态反思) / Reflexion      │     无树搜索，无多路径探索
                    ├──────────────────────────────────────┤
   L2 Planning      │ Plan-and-Solve / ADaPT / GoalAct    │  ← v1/v2 是 Plan-and-Solve
                    │ Backward Planning (v8)              │     v8 是 Backward Planning
                    ├──────────────────────────────────────┤
   L1 Execution     │ ReAct / CodeAct / Tool Calling      │  ← 项目是 ReAct
                    └──────────────────────────────────────┘     未引入 CodeAct
```

### 各层源码映射详情

**L1 执行层 — 已有 ReAct，缺 CodeAct**

- `agents/executor.py` — `_react_loop()` 是标准 ReAct（Thought → Action → Observation 循环）
- `react/engine.py` — v6 统一 ReActEngine（`ENABLE_REACT_ENGINE_V2` 特性开关）
- `tools/` 目录下有 `code_executor.py` + `shell_tool.py` — 沙盒执行基础设施齐全，但未被 LLM 直接用作 CodeAct 的"代码即动作"通道
- 工具调用方式：JSON function calling（`tools/base.py:to_openai_tool()` 生成 schema），LLM 每轮只能调一个或一组工具，无法在单轮中编写循环/条件/组合逻辑

**L2 规划层 — 四种范式并存**

- v1: `PlannerAgent.create_plan()` → `Plan(steps=[...])`（扁平线性）
- v2: `PlannerAgent.create_dag()` → `TaskDAG`（三层 hierarchy + 拓扑排序 + 超步执行）
- v5: `EmergentPlannerAgent` 动态 TodoList（涌现式，Claude Code 风格）
- v8: `GoalDrivenPlannerAgent` 目标文档 + 逆向规划（ReflAct 思想）
- 路由：`PlannerAgent.classify_task()` 两阶段混合分类器（规则快筛 60-70% + LLM fallback）

**L3 推理层 — 有线性反思，无搜索**

- v8 `_goal_reflect()`：每轮执行前对比 (current_state, goal_gap, next_step)，是 ReflAct 的直接实现
- v8 `_reanchor_goal()`：每 N 轮重新锚定目标文档，防止目标漂移
- v8 停滞检测：`_check_stagnation()` 检测连续无进度
- **缺失**：无 LATS/MCTS 树搜索——所有推理都是线性的，走进死胡同只能 replan 重来

**L4 协作层 — 完全空白**

- OrchestratorAgent 内部是单线程 LLM 循环，不派生独立子 agent
- 没有"多专业 agent 协作"范式
- 没有 context isolation 机制——子任务的全部中间过程堆积在父 agent 的 `_messages`
- `context/manager.py` 用 token budget + truncation 做压缩，是"截断"而非"隔离"

**L5 自进化 — 完全空白**

- `memory/long_term.py` 仅存 JSON KV，无"经验抽象"
- 无 skill library，无过程性记忆沉淀
- 任务完成后"学到的经验"完全丢失

## 三、架构约束与范式引入的关系

调研报告提出 v9/v10/v11 路线图，但需要从源码角度评估每个范式引入时受哪些架构约束影响：

| 范式 | 受影响的源码模块 | 架构约束 | 能否以特性开关叠加 |
|------|----------------|---------|-----------------|
| **S1 CodeAct** | `agents/executor.py`, `react/engine.py`, `tools/` | 需要新的 LLM 输出解析（Python 代码块 vs JSON tool_calls），现有 `think_with_tools()` 不支持 | ✅ 可以——新建 `agents/codeact_executor.py`，开关 `ENABLE_CODE_ACT` |
| **S2 LATS** | 新增 `agents/lats_planner.py` | 需要树结构数据模型 + UCT + LLM-as-Judge 评分，与现有 `schema.py` 正交 | ✅ 可以——纯新增模块 |
| **S3 CoA** | `agents/orchestrator.py`, `context/manager.py` | 需要 Worker-Manager 角色区分 + 上下文分段传递，与 Orchestrator 单线程模型冲突 | ⚠️ 需要改造 Orchestrator |
| **A1 MUSE** | `memory/` 目录 | 需要分层记忆 + embedding 检索，现有 `long_term.py` 太简单 | ⚠️ 需要引入 embedding 依赖 |
| **A2 Critic** | `agents/reflector.py`, `agents/goal_driven_planner.py` | 需要独立 prompt 的 critic agent，与现有 ReflectorAgent 角色 overlap | ✅ 可以——新建 `agents/critic.py` |
| **Subagent** | `agents/orchestrator.py`, `tools/`, `dag/executor.py` | 需要独立 agent 实例创建 + 生命周期管理 + 上下文隔离，与共享 ExecutorAgent 设计冲突 | ⚠️ 需要重构 agent 实例管理 |

## 四、两个正交维度的配置体系

当前项目已有两个正交维度：

| 维度 | 控制什么 | 配置项 |
|------|---------|--------|
| **规划调度层**（上层） | 任务怎么拆解、怎么调度 | `PLAN_MODE` + `EMERGENT_PLANNING_ENABLED` + `ENABLE_GOAL_DRIVEN_PLANNER` |
| **执行引擎层**（底层） | 每一步用什么实现跑 ReAct 循环 | `ENABLE_REACT_ENGINE_V2` |

上层决定"做什么、以什么顺序做"，底层决定"每一步用什么实现跑"。两个维度自由组合。

如果引入更多范式，配置体系会扩展为：

```
维度 1: 规划调度（已有）
  PLAN_MODE=simple/complex/emergent/auto

维度 2: 执行引擎（已有）
  ENABLE_REACT_ENGINE_V2=true/false

维度 3: 执行方式（如果引入 CodeAct）
  ENABLE_CODE_ACT=true/false

维度 4: 推理策略（如果引入 LATS）
  ENABLE_LATS=true/false

维度 5: 协作方式（如果引入 Subagent）
  ENABLE_SUBAGENT=true/false

维度 6: 校验方式（如果引入 Critic）
  ENABLE_CRITIC_VERIFIER=true/false
```

这些维度之间的正交性：
- CodeAct 与 LATS：**正交**（CodeAct 是执行层，LATS 是推理层）
- CodeAct 与 Subagent：**正交**（子 agent 内部可以用 CodeAct 或传统 ReAct）
- LATS 与 Subagent：**部分重叠**（LATS 可以在子 agent 内部运行）
- Critic 与所有其他：**正交**（独立的校验环节）
