# v9+ 融入路线图与实验设计

> **核心思路**: 不推翻 v1-v8，按"**每个 v 版本融入一个前沿范式 + 一组对照实验**"的教学节奏推进。
> 每个版本的设计都要回答 3 个问题：
> 1. 引入什么新范式？（学术来源）
> 2. 与现有版本的正交/替代关系？（特性开关位置）
> 3. 如何在 `evaluation/` 里做对照实验？（度量指标）

---

## 路线图总览

```
   v8 (当前)    ReflAct 目标驱动                   ← 已完成
       ↓
   v8.1        快速修补: 独立 Critic 双核校验        ← 小步快跑
       ↓
   v9          执行革命: CodeAct + LATS 树搜索      ← 主版本
       ↓
   v10         协作革命: Subagent + CoA + MUSE     ← 主版本
       ↓
   v11         Model-native 对照: R1/o3 直通       ← 压轴对照组
       ↓
   v12         Self-evolving 闭环                  ← 展望
```

## v8.1 — 独立 Critic 双核校验（快速修补，1 天工作量）

### 动机

Snorkel AI (2025) 证明 v8 的 `_goal_reflect`（同模型自我批评）存在 "Self-Critique Paradox"，在简单任务上反而降低性能。

### 实施

**新增文件**: `agents/critic.py` (~150 行)

```python
class CriticAgent(BaseAgent):
    """独立 prompt 的 critic，只负责验证：
    1. goal_doc.success_criteria 是否真的被满足
    2. 是否存在"虚假完成"（TODO 都勾了但目标没达到）
    3. 是否存在循环/停滞信号
    不参与任何行动，只返回 {'verdict': 'ok'/'block', 'reason': ...}
    """
```

**改造位置**: `agents/goal_driven_planner.py`
- 在 `_should_complete` 前增加 `critic.verify(goal_doc, todo_list)` 调用
- 若 critic 返回 block，则不允许声明完成，强制进入 reanchor

**配置**: `config.py` 新增 `ENABLE_CRITIC_VERIFIER=false`（默认关闭，研究时开启）

### 对照实验

| 条件 | 预期 |
|------|------|
| v8 baseline（关闭 critic）| 简单任务成功率 80%（含虚假完成） |
| v8 + critic | 简单任务成功率 82-85%（拦住虚假完成） |
| 成本对比 | LLM 调用增加约 15% |

## v9 — 执行革命（CodeAct + LATS）

### v9 = v8 + **CodeAct 执行层升级** + **LATS 树搜索推理**（可独立开关）

---

### 特性 A: CodeAct 执行层（S1 优先级，ROI 最高）

#### 动机

业内主流 agent（Manus / OpenManus / Claude Code / M365 Copilot）全部放弃 JSON tool-calling，改用 CodeAct。本项目作为教学 Demo 不能落伍。

#### 实施

**新增文件**:
- `agents/codeact_executor.py` (~400 行)
- `tools/code_action_sandbox.py` (~200 行) —— 复用 `tools/code_executor.py` + `tools/shell_tool.py` 作为底座

**核心改造**:
```python
class CodeActExecutor(BaseAgent):
    """
    每轮:
      Thought: LLM 输出推理 + Python 代码块
      Code:
        ```python
        result = web_search("X")
        filtered = [r for r in result if ...]
        return filtered
        ```
      Observation: stdout / stderr / return_value / exception
    """
    async def execute(self, task: str, tools_schema: dict):
        while not done:
            code_block = await self.think_code(task, history, tools_schema)
            obs = await self.sandbox.run(code_block)
            history.append((code_block, obs))
```

**工具 schema 改造**: 把所有 `tools/*.py` 的 function 签名注入到 code sandbox 的 namespace，让 LLM 输出的代码可以直接 `web_search(...)`, `read_file(...)` 等。

**配置**: `ENABLE_CODE_ACT=false`（默认关闭）

#### 对照实验（evaluation/benchmark.py 新增）

| 基准任务 | v5 (JSON ReAct) | v8 (JSON + ReflAct) | v9 (CodeAct + ReflAct) |
|---------|---------------|---------------------|----------------------|
| 12 个标准任务平均轮次 | ~12 | ~15 | **~6** (预期) |
| 成功率 | ~70% | ~78% | ~82% (预期) |
| Token 消耗 | 1x | 1.3x | **~0.6x** (预期) |

**教学亮点**: 可以在同一个任务 trace 里看到"3 轮 JSON 工具调用 vs 1 轮 CodeAct 代码块"的直观对比。

### 特性 B: LATS 树搜索推理（S2 优先级）

#### 动机

v8 是**线性反思**，在复杂任务上可能走进死胡同。LATS 用 MCTS 同时探索多条路径，显著提升 HumanEval/WebShop 成功率。

#### 实施

**新增文件**: `agents/lats_planner.py` (~800 行)

**核心数据结构**:
```python
@dataclass
class TreeNode:
    state: TodoList               # 复用 v8 的 TodoList
    action: Optional[str]          # 从父节点到此节点的 action
    value: float = 0.0             # UCT 打分
    visits: int = 0
    children: List["TreeNode"] = field(default_factory=list)
    reflection: Optional[GoalReflection] = None  # ← 融合 v8 ReflAct！
```

**四大阶段**:
1. **Selection**: UCT(`value/visits + c*sqrt(ln(N)/visits)`) 选叶子
2. **Expansion**: 对叶子生成 `TREE_WIDTH` 个候选 action (复用 v5 的 emergent_planner 生成 TODO 逻辑)
3. **Evaluation**: 对每个新节点做 ReflAct 式状态反思 + LLM-as-Judge 打分（**本项目的创新融合点**）
4. **Backpropagation**: 得分回传更新所有祖先的 UCT 统计

**配置**: `ENABLE_LATS=false`, `TREE_WIDTH=3`, `TREE_DEPTH=5`

#### 对照实验

| 基准任务 | v8 线性反思 | v9 LATS 树搜索 |
|---------|-----------|---------------|
| 12 个任务成功率 | ~78% | ~85% (预期) |
| LLM 调用量 | ~50 | **~150** (3x，可接受) |
| 难任务（多路径探索类）成功率 | 50% | **80%** (预期) |

**教学亮点**:
- Tracing Bridge 扩展：把 LATS 搜索树**可视化**到 Web Viewer（这是纯 LLM 可视化很少见的 demo）
- 可设计"对同一道逻辑推理题"的 v8 vs v9 trace 对比，直观展示"线性 vs 搜索"差异

---

### v9 两大特性的交叉验证

|  | CodeAct OFF | CodeAct ON |

## v10 — 协作革命（Subagent + CoA + MUSE）

### v10 = v9 + **Subagent 机制** + **Chain-of-Agents 长上下文协作** + **MUSE 经验沉淀**

---

### 特性 A: Subagent 机制（对标 Claude Code）

#### 动机

目前 `orchestrator.py` 是单线程 LLM 循环，上下文爆炸只能靠 truncation。Claude Code 的 Subagent 机制通过"派生子 agent + 独立 context + 只回传摘要"实现分形递归，是工业界最成熟的 context engineering 方案。

#### 实施

**新增工具**: `tools/spawn_subagent.py` (~150 行)

```python
class SpawnSubagentTool(BaseTool):
    """
    主 agent 调用:
      spawn_subagent(
        task="读取 docs/ 下的 5 个文档，总结各自要点",
        agent_type="emergent" | "goal_driven" | "codeact"
      )
    返回: 子 agent 完成后的摘要（不含完整历史）
    """
```

**改造**: `orchestrator.py` 的 `_execute_*` 方法都支持作为 subagent 被调用，完成后用独立的 summarize prompt 压缩输出。

**配置**: `ENABLE_SUBAGENT=false`, `MAX_SUBAGENT_DEPTH=3`

---

### 特性 B: Chain-of-Agents 长上下文协作

#### 动机

本项目 `context/manager.py` 目前仅用 token budget + truncation 处理长上下文，信息丢失严重。CoA 在长文档任务上能超越 RAG 10%+。

#### 实施

**新增文件**: `agents/chain_coordinator.py` (~500 行)

```python
class ChainCoordinator:
    """Manager-Worker 模式:
    1. split_task(task) → 分段 [chunk_1, ..., chunk_N]
    2. for chunk_i: cu_i = Worker.extract(chunk_i, prev_cus=[cu_1..cu_{i-1}])
    3. final_answer = Manager.synthesize(cus=[cu_1..cu_N])
    """
```

**改造**: v5 的 TODO 列表天然可以映射为 CoA 的 chunks，每个 TODO 由一个 Worker subagent 执行（与特性 A 自然结合）。

**配置**: `ENABLE_CHAIN_OF_AGENTS=false`, `CHAIN_WORKER_TYPE=codeact`

---

### 特性 C: MUSE 分层记忆

#### 动机

本项目 `memory/long_term.py` 只是 KV，完成任务后"学到的经验"完全丢失。MUSE 三层记忆在长时程任务上 SOTA。

#### 实施

**改造 `memory/` 结构**:
```
memory/
├── short_term.py       # 已有
├── long_term.py        # 已有，改造为入口
├── strategic.py        # 新增：高层策略（哪类任务用哪条路径）
├── tactical.py         # 新增：中层方法（工具组合套路）
├── episodic.py         # 新增：成功/失败轨迹存储 + 检索
└── memory_composer.py  # 新增：任务完成后的经验抽取
```

**改造 `agents/reflector.py`**: 任务完成后调用 `memory_composer.extract_and_store(trace, outcome)`，把成功模式升级到 tactical/strategic。

**检索**: `orchestrator._execute_emergent` 开始前，按 task embedding 检索相关 episodic/tactical memory 注入到 prompt。

**配置**: `ENABLE_MUSE_MEMORY=false`

#### 对照实验

**"重复任务"实验**: 同一个评测任务连续执行 5 次

| 次数 | v8 baseline 轮次 | v8 + MUSE 轮次 |
|------|-----------------|----------------|
| 第 1 次 | 15 | 15 |
| 第 2 次 | 15 | **8** (检索到第 1 次经验) |
| 第 3 次 | 15 | **6** (经验进一步固化) |
| 第 4 次 | 15 | **5** |
| 第 5 次 | 15 | **5** |

**教学亮点**: 可视化展示"agent 真的在学习"，这是自进化研究最吸睛的 demo。

## v11 — Model-native 对照（压轴章节）

### v11 = v10 + **ENABLE_MODEL_NATIVE 直通路径**

#### 动机

Sang et al. 2025 综述指出 agentic AI 正在从 Pipeline-based 走向 Model-native。本项目 v1-v10 全是 Pipeline-based（外部编排 894+ 行代码），**必须加一条"把任务+工具扔给 R1/o3 一把梭"的对照路径**，否则教学 Demo 脱离业界前沿。

#### 实施

**新增文件**:
- `llm/reasoning_client.py` (~80 行) —— 接入 DeepSeek-R1 / QwQ-32B / o3-mini API
- `agents/model_native.py` (~150 行)

```python
class ModelNativeAgent(BaseAgent):
    """
    完全不做外部规划/反思/编排。
    只做一件事：把 task + tools_schema 打包扔给 reasoning model，收 final_answer。
    """
    async def execute(self, task: str):
        return await self.reasoning_client.reason_with_tools(
            task=task,
            tools=get_all_tools_schema(),
            max_reasoning_tokens=32000  # R1 用，允许长 CoT
        )
```

**配置**: `ENABLE_MODEL_NATIVE=false`, `REASONING_MODEL="deepseek-r1"`

#### 压轴对照实验（评估报告中最重要的一张表）

| 维度 | v8 Pipeline (894 行) | v11 Model-native (230 行) |
|------|---------------------|--------------------------|
| 代码复杂度 | 极高 | 极低 |
| LLM 调用次数 | 30-80 次 | **1 次**（内部 long CoT） |
| 可观测性 | 极好（每步 event） | 差（reasoning 不透明） |
| 成本 | 低（小模型即可） | 高（需 reasoning model） |
| 12 任务成功率 | ~85% | **?** (待实验) |
| 长时程任务（5+ step） | ~70% | **?** (R1 擅长，可能碾压) |
| 控制力 | 强 | 弱 |

**教学结论**（预期）：
- 短期任务：Pipeline 仍具竞争力（便宜、可控）
- 长期任务：Model-native 大幅领先（内化的 long CoT 优势）
- **业界趋势**：能力内化后 Pipeline 会收缩为"工具胶水"角色

## v12 展望 — Self-evolving 闭环

不在本轮路线图，但留作未来方向：
- **自动 skill 抽取**: 从多次任务中归纳可复用的工具组合 → 注册为新"复合工具"
- **RL self-improve**: 用 v7 评测框架产生的 reward 信号做 offline RL（Voyager 思路）
- **Active exploration**: Agent 主动提出"我想学 X"并执行探索任务

## 对照实验的统一度量体系

所有新版本都要在 `evaluation/` 里输出一张对照表，度量以下 7 维：

| 维度 | 度量方式 |
|------|---------|
| 成功率 | 任务最终输出是否符合预期 |
| LLM 调用次数 | 从 Tracing Span 聚合 |
| Token 总消耗 | input + output 加总 |
| 平均轮次 | 到达 final answer 的 agent loop 数 |
| 失败恢复率 | 前 N 轮失败后最终仍成功的比例 |
| 可观测性分数 | event 覆盖度 / span 深度 |
| 上下文效率 | 峰值 context 使用量 / 任务复杂度 |

把这张表作为 **每个 v 版本的 README 标配**，让教学 Demo 的每次升级都有**量化的前后对比**。

---

## 总结：升级路线图的三个设计原则

1. **正交叠加而非替代**: 每个新范式通过特性开关引入，保留所有历史版本作为对照。
2. **一版本一重点**: v9 只碰执行层 + 推理层，v10 只碰协作 + 记忆，不求一次性大跃进。
3. **每版本都有对照实验**: 没有对照实验的版本不算完成——教学 Demo 的核心价值是"让读者看见差异"。

按此路线图，本项目将从 "v8 ReflAct 单点演示" 升级为 "**2024-2026 前沿 Agent 范式的系统性对照实验平台**"，在开源教学项目中独具定位。
