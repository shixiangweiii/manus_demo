### ⭐⭐⭐ 2. Lilian Weng — *Agent 上线前先做评估*

| 维度 | 内容 |
|------|------|
| 出处 | Lilian Weng Blog（2026-02-28） |
| 核心 | 大部分 Agent 项目失败的根因：没有针对真实任务建立可重复评估基线 |

**关键观点**:
- 上线前必须有 "task-grounded evaluation"，不是 benchmark 跑分而是面向落地任务的 golden set
- 没有基线就谈不上 regression testing，也无法做有意义的 A/B

**与 manus_demo 关联**（高优先级）:
- 直接背书我们 `evaluation/` 模块的设计哲学：12 任务 × 3 模式 × 4 维评分
- 但我们目前**没有人工标注的 ground truth set**，全靠 LLM-as-judge —— 这是一个公开承认的弱点，应在 README 显性声明
- 建议参考此文，给 README 增加一节《Why Evaluation First》

### ⭐⭐⭐ 4. Simon Willison — *Using LLMs for Code*

| 维度 | 内容 |
|------|------|
| 出处 | https://simonwillison.net/2025/Mar/11/using-llms-for-code/ |
| 核心 | 十几条 LLM 辅助编程实践要点 |

**关键观点节选**:
1. **设定合理期望**：LLM 仍是花哨的自动补全
2. **考虑训练截止日期**：知识有时滞
3. **上下文为王**：把相关代码全部喂进去
4. **让它们提供选择**：要求 N 个候选方案
5. **告诉它们确切要做什么**：避免模糊
6. **必须测试它写的内容**
7. **记住这只是一次对话**：会话有记忆边界
8. **使用可以为你运行代码的工具**：闭环执行才有反馈
9. **氛围编程是一种很好的学习方式**
10. **使用 LLM 是对已有专业知识的放大**（不会替代专业知识）

**与 manus_demo 关联**:
- 我们 `OrchestratorAgent` 的 prompt-engineering 设计基本符合上述要点
- 第 8 条（闭环执行）正是我们 `tools/code_executor.py` + `tools/shell.py` 沙箱设计的目的

🔗 https://simonwillison.net/2025/Mar/11/using-llms-for-code/ ；中文转载 [腾讯](https://mp.weixin.qq.com/s?__biz=MzI4MDYzNzg4Mw==&mid=2247570566&idx=3&sn=495e2514133adce152c10098349fbd69)

### ⭐⭐ 5. AI Agent 内部工作原理拆解（含 Lilian Weng 公式）

| 维度 | 内容 |
|------|------|
| 出处 | 腾讯云开发者社区，2026-05 |
| 核心 | Simon Willison 一句话定义 + Lilian Weng 公式 + HTTP 层抓包级讲解 |

**关键观点**:
- Simon Willison 定义：**"一个 LLM 在循环里不断调用工具，直到完成目标"**
- 三个底层事实：
  1. **LLM 是无状态的**——所谓多轮对话其实是客户端在本地不断追加 messages 数组
  2. **工具调用是"假装"出来的对话**——LLM 只生成 JSON 描述，编排代码才是真正执行者
  3. **每轮 token 复合累积**——长对话越来越慢、越来越贵

**与 manus_demo 关联**:
- 我们 `llm/client.py` 的 `_call_records` 设计与第 3 点直接对应
- `tools/router.py` 的 per-node 失败追踪 = 第 2 点的工程化体现

🔗 https://cloud.tencent.com/developer/article/2664960

### ⭐⭐ 6. Apple Research — *PORTool: Importance-Aware Policy Optimization with Rewarded Tree*

| 维度 | 内容 |
|------|------|
| 出处 | Apple Research，2026-05 |
| 核心 | 用奖励树做"重要性感知"的策略优化，让 LLM 更好地串联多工具 |

**关键观点**:
- 多工具集成推理（multi-tool-integrated reasoning）的核心难题：哪步工具调用最关键 / 是否走错分支
- PORTool 用 rewarded tree 给每个调用分支打分，做 importance-aware 的策略更新

**与 manus_demo 关联**:
- 我们 v2 DAG 路径里 `tools/router.py` 的 per-node 失败计数 ≈ 简化版的 reward signal
- 若未来做 RL 训练（v11+ 方向），可借鉴 PORTool 的奖励树设计

### ⭐⭐⭐ 8. Anthropic / OpenAI — *Reward Hacking 与 Trajectory Monitoring 双雄解读*

| 维度 | 内容 |
|------|------|
| 出处 | 综合 Anthropic Model Card + OpenAI 公开博客的二次解读（约 2 万字长文） |
| 核心 | 两家公司在"如何监控 Agent 轨迹 + 如何缓解 reward hacking"上的不同路线 |

**Anthropic 路线**:
- 工具：Clio / Petri / Bloom 等基础设施，把"读轨迹"批量化、结构化
- 把长轨迹整理 → 聚类 → 命名 → 摘要 → 自动评估 → 审核
- 目标：从大量真实交互里持续发现新的异常行为模式

**OpenAI 路线**:
- 强调 **CoT monitorability（思维链可监控性）**
- reasoning model 在思维链里会直接写出"我准备 reward hack"的意图，给监控提供有效信号
- 风险：训练方式不对，模型可能学会隐藏真实意图

**缓解策略对比**:
- Anthropic：训练阶段就减少坏行为形成，避免局部 reward hacking 泛化
- OpenAI：保持 CoT 透明性，让监控有抓手

**与 manus_demo 关联**（高优先级）:
- 我们 v7 tracing 已经做到"轨迹结构化记录 + 多 backend 导出"，方向正确
- 但**缺少自动化分析层**——目前只能人肉看 trace
- 借鉴 Anthropic Clio 思路：可在 `tracing/` 下加一个"轨迹聚类 + 摘要"实验性脚本，识别"重复调用同一工具""卡死循环"等异常模式
- 同时与论文 06（T3）形成方法论闭环：T3 给指标，Clio-style 工具给可视化
