## 一、Anthropic — Effective Context Engineering for AI Agents

- **官方原文**:https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- **中文翻译**:https://zhuanlan.zhihu.com/p/1957222140714661519
- **类型**:范式级官方文章
- **重要程度**:⭐⭐⭐⭐⭐

### 核心论点

> 「上下文工程是提示词工程的自然演进。」

把工作重心从「写 prompt」扩展到「策划整个上下文窗口」。

### 上下文的全集

```
┌──────────────────── Context Window ────────────────────┐
│  ┌──────────────────────────────────────────────────┐  │
│  │ System Instructions(系统指令)                    │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ Tools / Tool Definitions                         │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ MCP (Model Context Protocol)                     │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ External Data / RAG Results                      │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ Message History                                  │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ Run State(任务状态、临时变量)                     │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ Long-term Memory                                 │  │
│  └──────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

### 工程实践要点

1. **迭代过程**:每次推理都要决策「这次应该带哪些 token」
2. **token 预算管理**:有限窗口里要选最有价值的信息
3. **状态周期性优化**:Agent 循环中定期 prune / compress / re-rank
4. **不要 dump everything**:无关信息会稀释注意力,造成 Lost in the Middle

### 与 manus_demo 的对应

| Anthropic 概念 | manus_demo 实现 |
|---|---|
| System Instructions | 各 Agent 的 system prompt |
| Tools | `tools/` 模块 |
| Message History | `memory/ShortTermMemory` 滑动窗口 |
| Long-term Memory | `memory/LongTermMemory` JSON 持久化 |
| Run State | `dag/TaskDAG` + `react/ReActEngine` 循环状态 |
| Context curation | `context/ContextManager` LLM 压缩 |

## 二、Anthropic — Prompt Engineering Guide(Claude 4 适配版)

- **解读**:https://blog.csdn.net/TYLZVS007/article/details/150470429
- **类型**:实操指南
- **适用模型**:Claude 4(Opus 4.x、Sonnet 4.x)广泛适用其他模型

### 何时适用 Prompt Engineering

> 「先判断问题的性质,再选方法。」

- ✅ **适合 Prompt Engineering**:输出质量、风格、结构、内容相关问题
- ❌ **不适合 Prompt Engineering**:速度、成本、模型基础能力 → 应改用模型选择 / 微调 / 算法替换

### 提示工程 vs 微调

| 方面 | Prompt Engineering | Fine-tuning |
|---|---|---|
| 资源 | 仅文本输入 | 高端 GPU + 大内存 |
| 周期 | 小时级 | 周级 |
| 适用 | 行为引导、格式控制 | 领域知识固化、风格统一 |
| 成本 | 低 | 高(可达 prompt 优化的 20 倍) |

### 9 大提示技巧

1. **使用提示模板和变量** — 可复用的结构化骨架
2. **示例(Few-shot)** — 给模型示范输入输出对
3. **明确具体指令** — 删除模糊词("好"、"合适")
4. **思维链(CoT)** — 显式要求逐步推理
5. **使用 XML 标签** — 用 `<task>`、`<context>`、`<example>` 划分提示结构
6. **角色提示(System Role)** — 通过 system 消息赋予 Claude 角色
7. **预填充回复(Prefill)** — Claude 独有,在 assistant message 开头加引导文字
8. **链式提示(Prompt Chaining)** — 把复杂任务拆成多步、逐步处理
9. **长文本提示技巧** — 重要信息放开头/结尾,避免 Lost in the Middle

### Claude 4 特殊最佳实践

1. **控制响应格式**:用 XML 标签或 JSON Schema
2. **利用 Thinking 能力**:对复杂任务开启 extended thinking
3. **优化并行工具调用**:支持多工具同时调用,减少往返
4. **减少代理编码中的文件创建**:Claude 4 倾向多文件,需提示约束
5. **增强视觉和前端代码生成**:多模态 + 直接 HTML/CSS 输出
6. **避免硬编码测试通过**:提示要求真实实现,不要欺骗测试

### XML 标签的特殊地位

Claude 模型对 XML 标签**特别敏感**(训练数据偏向),建议使用:

```xml
<task>
  分析以下代码并指出 bug
</task>

<context>
  这是一个 Python web 服务,使用 FastAPI 框架
</context>

<code>
  ```python
  ...
  ```
</code>

<output_format>
  以 JSON 输出 {"bugs": [...], "suggestions": [...]}
</output_format>
```

### Prefill 技巧(Claude 独有)

```python
# 让 Claude 强制以 JSON 开头
messages = [
    {"role": "user", "content": "分析这段代码"},
    {"role": "assistant", "content": "{"}  # 预填充
]
# Claude 会接着 { 继续输出 JSON
```

## 三、Google — Prompt Engineering White Paper

- **作者**:Lee Boonstra(Google)
- **版本**:第 7 版(已公开发布)
- **解读 1**:https://cloud.tencent.com/developer/article/2637120
- **解读 2(25000 字详解)**:https://blog.csdn.net/weixin_55154866/article/details/156396266
- **类型**:系统教材级白皮书
- **重要程度**:⭐⭐⭐⭐⭐

### 文档结构(总目录)

| 章节 | 内容 |
|---|---|
| 简介 | Prompt Engineering 概述 |
| 大语言模型输出配置 | Output length / Temperature / Top-K / Top-P |
| **提示词技巧** | Zero-shot / Few-shot / System / Role / Context / Step-back / CoT / Self-consistency / ToT / ReAct / APE |
| **代码提示词** | 编写 / 解释 / 翻译 / 调试 |
| 多模态提示 | 图像 + 文本 |
| **最佳实践** | 10 余条工程经验 |

### 核心提示技术(11 类)

#### 1. Zero-shot Prompting(零样本)
直接给指令,无示例。适合简单、明确的任务。

#### 2. One-shot / Few-shot Prompting(单样本/少样本)
给 1-5 个输入输出对作为示例。
**关键经验**:示例质量 > 数量;错误示例会让性能下降 50%+。

#### 3. System Prompting
通过 system 消息设定全局规则、角色、约束。

#### 4. Role Prompting(角色提示)
让模型扮演特定角色("你是一名资深安全工程师")。

#### 5. Contextual Prompting(上下文提示)
在 prompt 中加入相关背景信息。

#### 6. Step-back Prompting(退步提示)
**先问一个更抽象的问题,再回到具体问题**。
例:不直接问「这个算法的时间复杂度?」,先问「分析时间复杂度的一般方法是什么?」,再让模型应用。

#### 7. Chain of Thought(CoT)
显式要求逐步推理,经典 prompt:`Let's think step by step.`

#### 8. Self-Consistency(自我一致性)
多次采样 + 多数投票。提升复杂推理任务准确率。

#### 9. Tree of Thoughts(ToT)
把推理建模成树,允许 backtrack 和分支探索。

#### 10. ReAct(Reasoning + Acting)
推理与工具调用交错,manus_demo 已实现。

#### 11. Automatic Prompt Engineering(APE)
让 LLM 自己生成和评估提示。

### 输出参数配置经验

| 参数 | 范围 | 经验值 |
|---|---|---|
| **temperature** | 0-2 | 事实任务 0;创意 0.7-0.9;高度创意 1.0+ |
| **top_p** | 0-1 | 高精度 0.1;通用 0.95 |
| **top_k** | int | 通常配合 top_p,设 40-50 |
| **max_tokens** | int | 严格限制以控制成本 |

**重要原则**:**不要同时调 temperature 和 top_p** — 选一个调。

### 10 条最佳实践

1. **提供示例**:Few-shot 几乎总是更好
2. **设计简洁性**:删除冗余,一句话能说清的不要两句
3. **明确输出要求**:格式、长度、结构都要说
4. **优先使用指令而非约束**:`do X` 比 `don't do Y` 更有效
5. **控制最大 token 长度**:省钱且让模型聚焦
6. **在提示词中使用变量**:模板化、可测试
7. **尝试不同的输入格式与写作风格**:实验驱动
8. **混合不同分类任务示例**:Few-shot 时打乱标签顺序
9. **结构化输出(JSON / XML)**:可机读、可校验
10. **记录和迭代**:每次 prompt 改动版本化记录

## 五、对 manus_demo 的具体应用建议

### 改进方向 1:System Prompt XML 化

manus_demo 当前 system prompt 多为 markdown,可考虑 Claude/通用混合方案:
- Claude 调用时:加 XML 标签
- 其他模型:保留 markdown

### 改进方向 2:增加 Step-back Prompting 路径

`PlannerAgent` 在分类前可先做一次「这类任务通常怎么处理?」的退步提示,再做具体规划。

### 改进方向 3:Self-Consistency 用于 Reflector

`ReflectorAgent` 的 verdict 可以多次采样投票,提升判断稳定性(代价:token 翻倍,需 cost-benefit 评估)。

### 改进方向 4:APE 自动优化

利用 manus_demo 的 `evaluation/` 模块作为 fitness function,跑 APE 循环优化各 Agent 系统提示。
