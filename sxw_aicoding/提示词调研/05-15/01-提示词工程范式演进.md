# 01 — 提示词工程范式演进:从 Prompt 到 Context Engineering

> 来源:Anthropic 官方文章 + 知乎《Prompt Engineering 2026 — 系列导论篇》+《2026 年提示词工程进阶策略》
> 类型:范式综述 + 思维框架

---

## 一、范式转变:Prompt Engineering → Context Engineering

### 1.1 Anthropic 官方定义

> 「使用语言模型的过程,正从『找到合适的提示词和短语』转变为『回答一个更宏观的问题:**什么样的上下文配置最有可能驱动模型产生我们想要的行为?**』」

**Prompt Engineering**:通过编写和组织 LLM 指令来获得最佳结果,核心是「写好系统提示和用户提示」。

**Context Engineering**:在 LLM 推理过程中,针对 token(信息)的最佳集合进行**策划与维护**。范围包括:
- 系统指令 / System Prompt
- 工具(Tool definitions)
- 模型上下文协议(MCP)
- 外部数据 / RAG 检索结果
- 消息历史(Message history)
- 运行状态(Run state)
- 长期记忆(Long-term memory)

### 1.2 核心区别

| | Prompt Engineering | Context Engineering |
|---|---|---|
| 任务边界 | 一次性编写(离散) | 迭代过程(每次推理都要策划) |
| 操作对象 | 指令文本 | 整个上下文窗口的 token 集合 |
| 适用阶段 | 应用早期(单轮分类、生成) | Agent 时代(多轮推理、长时跨度) |
| 工程难度 | 措辞优化 | 状态管理 + 信息筛选 + 工具循环 |

**关键洞察**:Agent 在循环中运行时,会不断生成与下一轮推理相关的数据,这些信息必须被**周期性地筛选和优化**。上下文工程,正是从这片不断演化的信息海洋中,策划出应纳入有限上下文窗口内容的一门艺术与科学。

## 二、2025 推理革命:Test-Time Compute 时代

### 2.1 核心标志事件

- **DeepSeek-R1**(开源):证明 RL 推理路径的可行性,推理模型平民化
- **Claude Opus 4.5 / 4.6 / 4.7**:混合推理架构(可控制 thinking 长度)
- **OpenAI O3 / GPT-5**:固定推理模型,内置长链思考
- **Google Gemini 3 Pro**:多模态混合推理

### 2.2 三大瓶颈挑战

1. **幻觉问题(Hallucination)** —— 推理过程长 → 错误传播放大
2. **Token 级思考的局限性** —— 思考粒度太细,缺乏宏观规划
3. **长上下文失效(Lost in the Middle)** —— 中段信息被忽略,这是 Context Engineering 的核心战场

## 三、四大范式转变(2026 实战必读)

### 转变 1:从一次性查询 → 迭代对话

```
旧范式:
  用户 → 一条「完美 prompt」→ 模型 → 输出

新范式:
  用户 → 初稿 prompt → 模型 → 输出 →
  评估 → 修正上下文 → 再次推理 → 收敛到目标
```

**实践含义**:不再追求「一次说对」,而是设计可迭代的上下文流。

### 转变 2:从通用策略 → 模型特定策略

不同模型架构需要不同的提示策略:

| 类型 | 代表模型 | 提示策略 |
|---|---|---|
| **混合推理** | Claude Opus 4.5/4.7、Gemini 3 Pro | 显式开启/关闭 thinking;提示直接说明任务 |
| **固定推理** | O3、DeepSeek-R1 | 不要写 CoT 提示,模型自带;提示要简洁 |
| **传统快速** | Qwen-Instruct、DeepSeek-V3.2 | 显式 CoT 提示、Few-shot 示例 |

### 转变 3:从参数规模 → 计算时间

模型能力的差异化竞争重点不再是「参数规模」,而是「Test-Time Compute(推理时计算)」。即给模型更多时间思考(更长的 thinking budget),效果可能比换更大参数模型更显著。

### 转变 4:从经验主义 → 科学方法

| 维度 | 经验做法 | 科学做法 |
|---|---|---|
| 理论基础 | 凭感觉 | 基于公开论文(ReAct、CoT、Lost in Middle) |
| 测试 | 人工试 | 系统化 Eval + 测试集 + A/B |
| 复现性 | 难复现 | 模板版本化、Seed 控制、记录上下文 |
| 优化 | 一次到位 | 持续监测 + 迭代优化 |

## 四、四类高频进阶策略

来源:《2026 年提示词工程进阶策略》

### 4.1 Expert Panel(多角色评审)

- **场景**:开放性、有争议、需要多视角的任务(如方案评估、决策建议)
- **机制**:让模型轮流扮演多个角色(产品经理 / 工程师 / 用户 / 法务等)对同一问题给出意见,最后综合
- **何时不适用**:简单事实问答、单视角任务 —— 多角色反而稀释焦点

### 4.2 Compression Protocol(压缩协议 / 关键信息锚点)

- **场景**:长文档摘要、长对话上下文压缩
- **机制**:用结构化标签(`<key>`、`<conclusion>`、`<evidence>`)锚定关键信息,避免 Lost in the Middle
- **理论支撑**:Lost in the Middle 论文显示,LLM 对上下文中段信息的利用率显著低于首尾

### 4.3 ReAct 工具循环

- **场景**:需要外部工具调用、需要实时信息的任务
- **机制**:Thought → Action → Observation 循环
- **关键论文**:Yao et al. 2022《ReAct: Synergizing Reasoning and Acting in Language Models》
- **manus_demo 实现参考**:`react/ReActEngine`

### 4.4 四层诊断框架

当 prompt 效果不佳时,**不要反复改句子**,而是按以下四层定位真正的故障点:

| 层级 | 检查项 | 典型修复 |
|---|---|---|
| **L1 成功标准** | 任务的「完成定义」清晰吗?可量化吗? | 补充 acceptance criteria |
| **L2 上下文配置** | 必须的信息进入上下文了吗?无关信息有没有干扰? | 增减上下文、调整顺序、加锚点 |
| **L3 工具循环** | 模型有没有正确的工具?循环何时终止? | 增减工具、调整 stopping condition |
| **L4 停止规则** | 模型怎么知道任务完成了?谁来验收? | 加 verification step、reflector |

## 五、Prompting 与 Context Engineering 的关系

```
              ┌─────────────────────────────┐
              │   Context Engineering       │ ← 2026 工作重心
              │  (策划进入上下文的所有token) │
              └────────────┬────────────────┘
                           │ 包含
                           ▼
       ┌──────────────────────────────────────────┐
       │ Prompt Engineering (措辞优化、结构化指令)  │ ← 仍是基础能力,但占比下降
       └──────────────────────────────────────────┘
                           │
                           ▼
                 ┌──────────────────┐
                 │ 简单单轮任务仍然有效 │
                 └──────────────────┘
```

## 六、对 manus_demo 项目的启示
