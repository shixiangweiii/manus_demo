## 一、可版本化的 PromptTemplate(强烈推荐)

- **来源**:CSDN《LLM Prompt 工程高级技巧 2026》
- **链接**:https://blog.csdn.net/yonggeit/article/details/160935849

### 核心思想

把 prompt 当代码:**版本号 + 必填变量 + 测试用例 + 渲染验证**。

### 参考实现(Python dataclass)

```python
from dataclasses import dataclass, field
from typing import Callable
from string import Template


@dataclass
class PromptTemplate:
    """可版本化、可测试的提示词模板。"""

    name: str
    version: str = "1.0"
    description: str = ""
    template: str = ""
    required_vars: list[str] = field(default_factory=list)
    optional_vars: dict[str, str] = field(default_factory=dict)
    test_cases: list[dict] = field(default_factory=list)

    def render(self, **kwargs) -> str:
        missing = [v for v in self.required_vars if v not in kwargs]
        if missing:
            raise ValueError(f"缺少必填变量:{missing}")
        variables = {**self.optional_vars, **kwargs}
        return Template(self.template).safe_substitute(variables)

    def validate(self, render_fn: Callable | None = None) -> list[str]:
        failures = []
        for case in self.test_cases:
            try:
                rendered = self.render(**case["input"])
                if "expected_contains" in case:
                    for expected in case["expected_contains"]:
                        if expected not in rendered:
                            failures.append(f"测试失败:渲染结果不包含 '{expected}'")
            except Exception as e:
                failures.append(f"渲染错误:{e}")
        return failures
```

### 工程价值

1. **必填变量校验** — 防止生产环境忘传 `${user_input}`
2. **可单元测试** — 模板就像函数,test_cases 是测试用例
3. **版本化** — `code_review_prompt.version = "2.1"`,可灰度发布
4. **可灰度** — 不同版本对照 A/B

### 对 manus_demo 的应用

当前 `agents/*.py` 中各 Agent 的 system prompt 是 Python 字符串字面量。可重构为:

```
prompts/
├── orchestrator_v1.0.json
├── planner_v2.3.json
├── executor_v1.5.json
└── reflector_v1.2.json
```

加载器统一从 JSON 读取并注入到 Agent。

## 二、Expert Panel(多角色评审)实操

- **来源**:知乎《2026 年提示词工程进阶策略》
- **链接**:https://zhuanlan.zhihu.com/p/2035846499507160429

### 适用判断

| 任务特征 | 是否适合 Expert Panel |
|---|---|
| 单一事实问答 | ❌ 不适合 |
| 简单代码生成 | ❌ 不适合 |
| 方案设计、架构选型 | ✅ 适合 |
| 法律/合规审查 | ✅ 适合 |
| 商业决策建议 | ✅ 适合 |
| 内容质量评审 | ✅ 适合 |

### 模板示例

```
你将依次扮演以下三个角色,对同一问题给出意见。

<角色1:产品经理>
关注用户体验和市场契合。
</角色1>

<角色2:技术架构师>
关注技术可行性和长期维护成本。
</角色2>

<角色3:风险合规官>
关注法律风险和合规边界。
</角色3>

<问题>
{问题}
</问题>

输出格式:
{
  "perspectives": [
    {"role": "PM", "view": "...", "concerns": ["..."]},
    {"role": "Architect", "view": "...", "concerns": ["..."]},
    {"role": "Risk", "view": "...", "concerns": ["..."]}
  ],
  "synthesis": "综合三个角色,我的最终建议是...",
  "trade_offs": ["..."]
}
```

### 注意事项

- **不要超过 4 个角色** — token 成本高,角色稀释焦点
- **角色之间要有冲突点** — 同质角色没价值
- **synthesis 是关键** — 单纯罗列不如有综合判断

## 三、Compression Protocol(关键信息锚点)

- **来源**:同上知乎文章
- **理论支撑**:Liu et al.《Lost in the Middle》论文

### 问题背景

LLM 对长上下文的注意力分布:**首段 > 末段 > 中段**。
中段信息可能被「遗忘」,这是长文档处理的最大陷阱。

### 解决方案:结构化锚点

```xml
<context>
<key id="K1" priority="high">
  这是后续推理必须用到的事实 1
</key>

<key id="K2" priority="high">
  这是后续推理必须用到的事实 2
</key>

<background>
  以下是一些背景信息(优先级低)
  ...大段文字...
</background>

<key id="K3" priority="high">
  这是后续推理必须用到的事实 3
</key>
</context>

<task>
  根据上述 context 中的 K1、K2、K3,回答...
</task>
```

### 关键技巧

1. **关键信息双锚定**:开头总结 + 末尾重申
2. **显式 ID 引用**:让模型在 reasoning 中引用 K1/K2,可监控注意力
3. **背景信息隔离**:用标签把「必读」和「可选」分开

### manus_demo 应用

`context/ContextManager._find_safe_split()` 已经处理了 tool_calls 边界安全问题,可以再加一层「关键信息提取并置顶」机制。

## 四、ReAct + Reflexion 闭环

- **来源**:多篇 CSDN 博客综述
- **manus_demo 已部分实现**:`react/ReActEngine`

### ReAct(已实现)

```
Thought → Action → Observation → Thought → ...
```

### Reflexion 增量(可加)

ReAct 完成一轮任务后,加一次**自我反思**:

```python
async def reflect(self, task, trajectory):
    """反思整个 ReAct 轨迹,生成经验教训。"""
    prompt = f"""
    任务:{task}
    执行轨迹:{trajectory}

    请反思:
    1. 任务是否完成?(yes/no/partially)
    2. 哪一步出现了问题?
    3. 下次遇到类似任务,应该如何改进?
    4. 把改进存到 memory 中作为后续任务的提示
    """
    return await self.llm.complete(prompt)
```

### Reflexion vs Reflector(manus_demo 已有)

| | manus_demo Reflector | Reflexion 论文版 |
|---|---|---|
| 时机 | 任务完成后 | 任务完成后 + 失败重试前 |
| 输出 | 通过/失败判断 | 通过/失败 + 经验教训写回 memory |
| 后续 | 触发 replan | 影响下一次推理的 prompt |

可以让 manus_demo 的 Reflector 在判断「失败」时,把经验写入 LongTermMemory,下次类似任务自动检索。

## 五、CLEAR 原则(Anthropic 推广)

- **来源**:Anthropic 文档汇总博客

| 字母 | 含义 | 反例 → 正例 |
|---|---|---|
| **C**oncise | 简洁 | "请你仔细思考后..." → "用不超过 50 字回答" |
| **L**ogical | 有逻辑 | 散乱要求 → 1. 2. 3. 列出 |
| **E**xplicit | 明确 | "好一点的代码" → "符合 PEP 8、类型注解完整、有 docstring" |
| **A**daptive | 可适配 | 写死场景 → 用变量 ${场景}、${语言} |
| **R**eflective | 可反思 | 一次定型 → 跑 eval、记录、迭代 |

## 六、四层金字塔结构(中文实战手册)

- **来源**:多篇中文实战博客综合

```
┌─────────────────────────────┐
│       L4: 输出格式约束       │  ← JSON Schema、字段、长度
├─────────────────────────────┤
│       L3: 任务说明详情       │  ← 步骤、优先级、边界条件
├─────────────────────────────┤
│       L2: 上下文与示例       │  ← 背景、Few-shot、参考资料
├─────────────────────────────┤
│       L1: 角色与目标         │  ← System role、长期目标
└─────────────────────────────┘
```

**自下而上构建,自上而下输入。**

### 实战模板

```
# L1 角色与目标
你是一个 10 年经验的 Python 后端工程师,目标是写出可维护的生产级代码。

# L2 上下文与示例
项目使用 FastAPI + SQLAlchemy + Pydantic v2。
代码风格示例:
```python
@router.get("/users/{user_id}")
async def get_user(user_id: int, db: AsyncSession = Depends(get_db)) -> UserOut:
    ...
```

# L3 任务说明
请实现一个 POST /users 接口:
1. 接收 UserCreate 请求体(email、password)
2. 校验 email 唯一性
3. 哈希密码后存库
4. 返回 UserOut(不含密码)

# L4 输出格式
- 仅输出代码块,不要解释
- 包含完整的路由、请求/响应模型、依赖注入
- 使用类型注解
```

## 七、负面清单策略(明确「不要做什么」)

```
❌ 错误示例:
写一篇文章,不要太长,不要太技术化,不要有废话。

✅ 正确示例:
写一篇 500 字左右的文章,用通俗语言,直接切入主题。
```

**原则**:
- **优先用正向指令**(`do X`)
- **必要时再用负向约束**(`don't Y`)— 仅在已经反复出错时使用
- 不要堆叠 5+ 个 "don't",模型会忽略其中一些

## 八、模型特定策略矩阵

| 模型类型 | 代表 | CoT 提示 | XML 标签 | Few-shot 数量 | Temperature |
|---|---|---|---|---|---|
| 混合推理 | Claude 4.x、Gemini 3 | 不需要(自带) | 强烈推荐 | 1-3 | 0.3 |
| 固定推理 | O3、DeepSeek-R1 | **不要写**(干扰) | 可选 | 0-1 | 0.5 |
| 传统快速 | Qwen-Instruct、DeepSeek-V3.2 | **强烈需要** | 可选 | 3-5 | 0.7 |

**误用警告**:在 O3 / DeepSeek-R1 这类自带推理的模型上写 `Let's think step by step` 反而会**降低**性能。

## 九、本节方法论速查表

| 方法 | 何时用 | 何时不用 | 一行代码示例 |
|---|---|---|---|
| Few-shot | 格式不稳定 | 简单事实问答 | 加 3 个 `<example>` 块 |
| CoT | 推理型模型(非自带 CoT) | O3/R1 自带推理模型 | `Let's think step by step` |
| Expert Panel | 决策、设计 | 事实问答 | 让模型轮流扮演 3 个角色 |
| Compression Protocol | 长文档/长对话 | 短任务 | 用 `<key>` 标签锚定 |
| ReAct | 需要工具/实时信息 | 纯推理 | Thought→Action→Obs 循环 |
| Reflexion | 多次失败重试 | 一次性任务 | 失败后反思写入 memory |
| Step-back | 抽象/原理性问题 | 直接事实 | 先问「一般方法」再问具体 |
| Self-Consistency | 数学/逻辑推理 | 创意写作 | 采样 5 次投票 |
