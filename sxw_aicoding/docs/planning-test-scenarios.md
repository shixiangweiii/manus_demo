# 规划测试场景 v5

> 本文档基于最新源码实现，测试三路由混合规划系统（Simple/Complex/Emergent）的自动路由能力和执行质量。

## 一、如何运行这些用例

### 1.1 交互模式

启动交互式 CLI：

```bash
python main.py
```

在提示符后直接输入用例中的任务描述即可。

### 1.2 单次任务模式

通过命令行参数直接执行单个任务：

```bash
PLAN_MODE=auto python main.py "任务描述"
```

### 1.3 强制路由模式（用于调试）

强制使用特定路由进行测试：

```bash
# 强制走 Simple v1 扁平计划
PLAN_MODE=simple python main.py "任务描述"

# 强制走 Complex v2 DAG
PLAN_MODE=complex python main.py "任务描述"

# 强制走 Emergent v5 隐式规划
PLAN_MODE=emergent python main.py "任务描述"
```

### 1.4 启用详细日志

```bash
python main.py "任务描述" --verbose
```

或简写：

```bash
python main.py "任务描述" -v
```

### 1.5 配置说明

关键配置项（通过 `.env` 文件或环境变量设置）：

- **PLAN_MODE**：规划模式
  - `auto`（默认）：自动路由（三路由混合分类器）
  - `simple`：强制走 v1 扁平计划
  - `complex`：强制走 v2 DAG
  - `emergent`：强制走 v5 隐式规划

- **EMERGENT_PLANNING_ENABLED**：是否启用 v5 隐式规划（默认 `true`）

- **ENABLE_REACT_ENGINE_V2**：是否使用 ReAct Engine v2（默认 `false`）

- **LLM_RETRY_ENABLED**：是否启用 LLM 重试机制（默认 `false`）

## 二、测试维度与观测点

### 2.1 路由正确性

**观测点**：

1. **分类器行为**：
   - 规则快筛（`_rule_classify`）是否正确识别明显简单/复杂/探索性任务
   - LLM 兜底分类（`_llm_classify`）是否仅在模糊区间触发
   - 终端是否显示 `Task complexity: Simple/Complex/Emergent`

2. **模式匹配**：
   - `_EXPLORATORY_PATTERN` 是否正确捕获探索性任务（如"探索"、"调研"、"分析并建议"）
   - `_UNCERTAINTY_PATTERN` 是否正确捕获不确定性任务（如"不确定"、"可能"、"尝试"）

3. **路由决策**：
   - Simple 任务是否走 v1 扁平计划（2-6 步线性步骤）
   - Complex 任务是否走 v2 DAG（Goal -> SubGoals -> Actions）
   - Emergent 任务是否走 v5 隐式规划（TODO 列表管理）

### 2.2 计划质量

**观测点**：

1. **Simple v1 路径**：
   - 步骤数量是否在 2-6 步之间
   - 步骤描述是否清晰、可执行
   - 步骤依赖关系是否合理
   - 终端显示 `Simple Plan (v1)` 表格

2. **Complex v2 路径**：
   - DAG 层级是否为 3 层（Goal -> SubGoals -> Actions）
   - SubGoals 数量是否为 2-5 个
   - 每个 SubGoal 是否包含 1-3 个 Action
   - 条件分支是否正确建模（`if`/`then` 逻辑）
   - 终端显示 `Task DAG` 树形结构

3. **Emergent v5 路径**：
   - TODO 列表是否动态生成和管理
   - 是否显示 `TODO List` 相关输出
   - 是否在执行过程中逐步添加/完成任务项

### 2.3 执行效率

**观测点**：

1. **执行速度**：
   - Simple 用例：秒级响应（主观感受）
   - Complex 用例：明显慢于 Simple，但可通过并行加速
   - Emergent 用例：根据探索深度变化

2. **并行执行**：
   - Complex 路径下，Super-step 中是否并行执行多个节点
   - 终端显示 `--- Super-step X --- Running Y/Z nodes (parallel)`

3. **资源消耗**：
   - LLM 调用次数和 token 消耗
   - 工具调用频率和成功率

### 2.5 错误处理与自愈

**观测点**：

1. **节点失败**：
   - 终端显示 `<< node_id FAILED (execution/exit_criteria).`
   - 是否触发局部重规划（仅重建失败子树）

2. **回滚机制**：
   - 终端显示 `<< node_id rolled back.`
   - 状态转移：`RUNNING -> FAILED -> ROLLED_BACK`

3. **重规划**：
   - 反思失败后是否触发 `Partial replan`
   - 新 DAG 是否保留已完成工作

### S1：查询上海今天天气

- **难度**：简单
- **任务描述**：
  - `查询上海今天天气`
- **运行方式**：
  - 交互模式输入上述任务。
  - 单次任务模式：`PLAN_MODE=auto python main.py "查询上海今天天气"`
- **推荐 PLAN_MODE**：`auto`
- **预期路由与计划形态**：
  - 规则分类应直接判为 **simple**（文本很短、动作单一）。
  - 走 **Simple v1 扁平计划**，计划包含约 1–2 个步骤（搜索 + 总结）。
- **关键观测点**：
  - 终端中显示：`Task complexity: Simple (v1 flat plan)`。
  - `Simple Plan (v1)` 表格中的步骤数与描述是否紧扣「查询天气」。
  - 工具调用中通常会使用 `web_search`（如果配置了真实搜索）或 mock 搜索。
- **评价标准**：
  - **规划正确性**：是否只包含必要的 1–2 步，无明显冗余步骤。
  - **执行速度**：整体耗时应非常短（主观感受为「秒级响应」）。

## 七、中等难度用例（边界与混合场景）

中等难度用例主要用于验证：

- 规则分类器 `_rule_classify()` 的边界行为。
- LLM 兜底分类 `_llm_classify()` 是否只在必要时触发，且给出合理判定。
- Simple / Complex 两条路径在同一任务规模下的结构与速度差异。

### M1：搜索异常处理最佳实践并整理注意事项

- **难度**：中等
- **任务描述**：
  - `先搜索 Python 异常处理的最佳实践，然后整理成 3 条注意事项`
- **运行方式**：
  - 交互模式输入上述任务。
  - 单次任务模式：`PLAN_MODE=auto python main.py "先搜索 Python 异常处理的最佳实践，然后整理成 3 条注意事项"`
- **推荐 PLAN_MODE**：`auto`
- **理想路径**：
  - 规则评分可能为 **ambiguous**，触发 `_llm_classify()`。
  - 理想情况下被判为 **simple**：2–4 步扁平线性计划即可覆盖需求。
- **预期计划形态**：
  - 步骤示例：搜索资料 → 过滤/阅读 → 总结出 3 条注意事项。
- **关键观测点**：
  - 是否在日志中看到「Rule classifier: ambiguous, invoking LLM classifier」。
  - `Task complexity` 最终被判为 Simple 或 Complex。
  - Simple 情况下，是否仍能完整覆盖检索 + 总结流程。
- **评价标准**：
  - **规划正确性**：无论路由到 Simple 还是 Complex，计划是否覆盖「搜索→整理」两个核心阶段。
  - **执行速度**：Simple 通常更快；如果被判为 Complex，应看到更详细分解，但延迟略高。
  - **结果质量**：输出的 3 条注意事项是否具体、有指导价值。

- **难度**：中等

- **推荐 PLAN_MODE**：`auto`
- **理想路径**：
  - 规则分类应给出 ambiguous，触发 LLM 兜底。
  - 理想情况下仍判为 **simple**，生成 3–5 步扁平计划（列出文件 → 过滤 → 计数 → 总结）。
