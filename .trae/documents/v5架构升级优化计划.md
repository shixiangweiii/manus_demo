## 二、发现的问题

### 2.1 代码重复问题 (Critical)

| 问题 | 位置 | 影响 |
|------|------|------|
| **ReAct 循环重复** | `emergent_planner.py:324-427` vs `executor.py:152-278` | ~120 行重复代码，维护成本高 |
| **TODO 列表类与 DAG 类职责重叠** | `schema.py:357-462` vs `dag/graph.py` | 两种任务表示方式，复杂度增加 |
| **节点状态机分散** | `NodeStateMachine` vs `TodoStatus` | 两套状态机，概念不统一 |

### 2.2 技术债务

| 问题 | 描述 | 优先级 |
|------|------|--------|
| Token 估算不精确 | 简单 `len/3` 估算，与实际 token 数偏差大 | Medium |
| LLM 调用无重试 | API 失败时直接抛异常，无容错 | High |
| 无标准错误框架 | 各 Agent 错误处理不一致 | Medium |
| 事件系统不完善 | `_emit()` 回调缺乏事件协议 | Low |
| 缺乏可观测性 | 无执行追踪、性能分析 | Medium |

### 2.3 架构扩展性问题

| 问题 | 描述 | 优先级 |
|------|------|--------|
| 工具生态受限 | 仅 web_search/file_ops/code_executor | High |
| 无插件机制 | 无法动态加载自定义 Agent/工具 | Medium |
| TODO 无检查点 | 长任务中断无法恢复 | Medium |
| Planner 职责过重 | ~400 行方法，功能耦合 | Low |

## 三、升级优化计划

### Phase 1: 核心重构（建议 v6.0）

#### 1.1 抽取 ReAct 执行引擎 ⭐⭐⭐

**目标**: 消除重复代码，统一 ReAct 执行逻辑

**实现方案**:
```
新建: react/engine.py
      - ReActEngine 类 (单例或依赖注入)
        - execute(prompt, tools, context) -> StepResult
        - 包含完整的 ReAct 循环逻辑
        - 集成 ToolRouter
        - 支持迭代控制、错误处理

修改:
  - ExecutorAgent: 委托给 ReActEngine
  - EmergentPlannerAgent: 委托给 ReActEngine
```

**预期收益**: 减少 ~120 行重复代码，单一维护点

#### 1.2 统一任务表示模型 ⭐⭐⭐

**目标**: 消除 DAG vs TODO 的概念分裂

**实现方案 A - 继承方案**:
```
TaskGraph (抽象基类)
  ├── DAGTaskGraph (继承)
  │     - 层级节点 (Goal/SubGoal/Action)
  │     - 边类型 (DEPENDENCY/CONDITIONAL/ROLLBACK)
  │     - 并行 Super-step 执行
  └── LinearTaskGraph (继承)
        - 扁平 TODO 列表
        - 顺序执行 + 依赖检查
```

**实现方案 B - 接口方案** (推荐):
```
TaskGraph (Protocol)
  - get_ready_items() -> list[TaskItem]
  - mark_completed(item_id, result)
  - is_complete() -> bool

DAGTaskGraph implements TaskGraph
EmergentTaskGraph implements TaskGraph
```

**预期收益**: 架构更清晰，可扩展性更强

#### 1.3 LLM Client 增强 ⭐⭐

**目标**: 添加容错、重试、限流机制

**实现方案**:
```python
class LLMClient:
    - retry_config: RetryConfig
      - max_retries: 3
      - backoff_factor: 2.0
      - retry_on: [RateLimitError, TimeoutError]

    - circuit_breaker: CircuitBreaker
      - failure_threshold: 5
      - recovery_timeout: 60s

    - rate_limiter: TokenBucket
      - requests_per_minute: 可配置
```

**预期收益**: API 稳定性提升，优雅降级

### 8.2 兼容性保障策略

#### 策略 1: 接口兼容封装

```python
# 旧接口保持不变，内部委托给新实现
class ExecutorAgent:
    """向后兼容封装 - 原有接口完全保留"""

    async def execute_step(self, step: Step, context: str = "") -> StepResult:
        # 旧路径：直接调用原有逻辑
        return await self._legacy_execute_step(step, context)

    async def _legacy_execute_step(self, step: Step, context: str) -> StepResult:
        # 原有的执行逻辑完整保留
        ...
```

#### 策略 2: Feature Flag 控制

```python
# 配置项控制新功能启用
config.py:
    ENABLE_REACT_ENGINE_V2 = os.getenv("ENABLE_REACT_ENGINE_V2", "false")
    ENABLE_UNIFIED_TASK_GRAPH = os.getenv("ENABLE_UNIFIED_TASK_GRAPH", "false")

# 运行时决策
class ExecutorAgent:
    def __init__(self, ...):
        self._use_new_engine = config.ENABLE_REACT_ENGINE_V2

    async def execute_step(self, step, context):
        if self._use_new_engine:
            return await self._react_engine.execute(...)
        else:
            return await self._legacy_execute_step(...)
```

#### 策略 3: 渐进式迁移

```
Phase 1: 新旧并存
├── ExecutorAgent 同时支持新旧执行路径
├── EmergentPlannerAgent 可选使用新引擎
└── Feature Flag 控制，默认关闭新功能

Phase 2: 新路径稳定后，默认启用
├── config.ENABLE_* 默认为 true
└── 旧路径仍可通过配置回退

Phase 3: 旧路径废弃（v7.0）
├── 移除旧路径代码
```

#### 8.3.4 事件系统标准化兼容性

**受影响组件**: UI 回调 `on_event`

**保障措施**:
1. `EventBus` 兼容原有 `on_event` 回调签名
2. `EventBus.emit(event, data)` 自动适配旧接口
3. 原有 `_emit()` 方法通过适配器桥接

```python
class EventBus:
    def __init__(self, legacy_callback=None):
        self._legacy = legacy_callback

    def emit(self, event, data):
        # 兼容旧接口
        if self._legacy:
            self._legacy(event, data)
        # 新接口
        for handler in self._handlers.get(event, []):
            handler(data)
```
